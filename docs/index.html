Overview

How It Works

Kova Auth uses Firebase Realtime Database as its backend. Your application connects directly to Firebase using credentials from your dashboard. The SDKs handle license validation, HWID locking, and expiry checking automatically.

Setup

App Credentials

When you create an application in the dashboard, you receive these credentials. All of them are required for SDK initialization.

CredentialTypeDescription
ownerIdstring (16 hex)Your unique owner identifier, found at the top of the dashboard
appNamestringApplication name — must match exactly as shown in the dashboard
appSecretstring (64 hex)Auto-generated secret key unique to each application
appVersionstringApplication version you set when creating the app (e.g., "1.0")
apiUrlstringFirebase Realtime Database URL — provided automatically in code examples
Tip: Click the Code button on any app card in the dashboard to get pre-filled integration code with all your credentials.
SDK

C# Integration

Initialize the Kova Auth client with your credentials, then call license() to validate a key.

// Initialize Kova Auth
var kova = new api(ownerId, appName, appSecret, appVersion, apiUrl);

// Prompt user for license key
Console.Write("Enter License Key: ");
string key = Console.ReadLine();

// Validate — first use locks to HWID automatically
kova.license(key.Trim());

if (kova.success)
{
    Console.WriteLine("Licensed! " + kova.message);
    Console.WriteLine("Expires: " + kova.expiryDate);
    Console.WriteLine("HWID: " + kova.hwid);
}
else
{
    Console.WriteLine("Failed: " + kova.message);
}
Download: Get the full C# SDK from the dashboard — click the Download button on your app card and select C#.
SDK

C++ Integration

Same flow — initialize with credentials, call license() to validate. Includes built-in anti-tamper via Security.h.

#include "KovaAuth.h"
#include "Security.h"

// Run anti-tamper protection
Security::Initialize();

// Initialize Kova Auth
KovaAuth::api kova(ownerId, appName, appSecret, appVersion, apiUrl);

std::cout << "Enter License Key: ";
std::string key;
std::getline(std::cin, key);

// Validate — first use locks to HWID automatically
kova.license(key);

if (kova.success) {
    std::cout << "Licensed! " << kova.message << std::endl;
} else {
    std::cout << "Failed: " << kova.message << std::endl;
}
SDK

DLL Integration (LoadLibrary)

Load KovaAuth.dll at runtime and call exported functions. Useful for injecting auth into existing projects.

HMODULE hDll = LoadLibraryA("KovaAuth.dll");

auto pInit     = (fnKovaInit)GetProcAddress(hDll, "KovaInit");
auto pValidate = (fnKovaValidate)GetProcAddress(hDll, "KovaValidate");
auto pGetMsg   = (fnKovaGetMessage)GetProcAddress(hDll, "KovaGetMessage");
auto pSuccess  = (fnKovaIsSuccess)GetProcAddress(hDll, "KovaIsSuccess");

// Initialize with your credentials
pInit(ownerId, appName, appSecret, appVersion, apiUrl);

// Validate license key
pValidate(key.c_str());

if (pSuccess()) {
    std::cout << "Licensed! " << pGetMsg() << std::endl;
}
DLL ExportSignatureDescription
KovaInitvoid(const char* x5)Initialize with ownerId, appName, secret, version, apiUrl
KovaValidateint(const char*)Validate a license key, returns 1 on success
KovaGetHWIDconst char*()Get the current machine's hardware ID
KovaGetMessageconst char*()Get the last status/error message
KovaIsSuccessint()Check if last validation succeeded (1 = yes)
Lifecycle

License Validation Flow

When a user enters a license key, the SDK performs these checks:

StepCheckResult
1Key exists in databaseIf not found → Invalid key
2Expiry date checkIf expiryDate < now → Expired
3Status is "Not used"First activation → HWID locked, status set to Used
4Status is "Used" + HWID matchSame device → Authenticated
5Status is "Used" + HWID mismatchDifferent device → HWID locked
HWID Locking: On first use, the license is permanently bound to the machine's hardware ID. Subsequent uses from a different device are rejected.
Reference

License Object

Each license stored in the database contains these fields:

FieldTypeDescription
keystringLicense key in format XXXX-XXXX-XXXX-XXXX-XXXX
typestringLicense type (e.g., "1 Day", "7 Days", "30 Days", "Lifetime", "Custom")
statusstring"Not used" or "Used"
expiryDatenumberUnix timestamp (ms) when the license expires
createdAtnumberUnix timestamp (ms) when the license was created
hwidstring | nullHardware ID bound on first activation
activatedAtnumber | nullUnix timestamp (ms) of first activation
notestringOptional note set during license creation

Advanced

Firebase Data Structure

For advanced integrations, here's how data is organized in Firebase Realtime Database:

├── users/
│   └── {uid}/
│       ├── username
│       ├── email
│       ├── ownerId          ← 16-char hex, used in SDK init
│       ├── plan             ← "free" or "premium"
│       ├── licenseLimit     ← max licenses (free plan = 5)
│       └── createdAt
│
├── applications/
│   └── {uid}/
│       └── {appId}/
│           ├── name         ← must match appName in SDK
│           ├── version
│           ├── secret       ← 64-char hex, must match appSecret in SDK
│           ├── ownerId
│           └── createdAt
│
└── licenses/
    └── {appId}/
        └── {licenseId}/
            ├── key          ← XXXX-XXXX-XXXX-XXXX-XXXX
            ├── type
            ├── status       ← "Not used" or "Used"
            ├── expiryDate   ← Unix timestamp (ms)
            ├── hwid         ← set on first activation
            ├── activatedAt  ← set on first activation
            ├── note
            └── createdAt
Security: Never expose your appSecret in client-side web code. The SDKs are designed for native desktop applications (C#, C++) where the secret is compiled into the binary.
